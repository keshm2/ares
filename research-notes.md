# Research — Calm Premium Desktop App UI Direction

## Summary
Use a neutral-first, grid-disciplined desktop shell with quiet surfaces, restrained accent color, and data hierarchy carried by size, spacing, contrast, and labels rather than decoration — this is consistent with Atlassian grid guidance and Carbon dashboard guidance: https://atlassian.design/foundations/grid, https://carbon-website-git-pictograms-docs.carbon-design-system.vercel.app/data-visualization/dashboards. Motion should become a routing primitive, not garnish: page changes should preserve context with short opacity/translate transitions or View Transition API-backed shared-element transitions where supported, while honoring Reduced Motion: https://developer.mozilla.org/en-US/docs/Web/API/View_Transition_API, https://developer.apple.com/help/app-store-connect/manage-app-accessibility/reduced-motion-evaluation-criteria/. Keep the current system-font default and bundled Geist option for implementation now, then evaluate Inter, Geist, IBM Plex, and Atkinson Hyperlegible as future directions based on dashboard density, tone, and accessibility needs: file:///Users/keshmuthu/aplyx/desktop/src/styles/tokens.css, https://rsms.me/inter/, https://vercel.com/font, https://www.ibm.com/design/language/typography/typeface/, https://www.brailleinstitute.org/freefont/.

## Approaches / Subtopics
### Dashboard / Layout Recommendations
- Use a stable desktop app frame: persistent left navigation, one main content column, and a small number of top-level content containers aligned to a grid; Atlassian explicitly scopes grid alignment to main content containers such as cards, tables, forms, and text blocks, while small controls should rely on spacing tokens instead: https://atlassian.design/foundations/grid.
- Prioritize the screen by importance: Carbon recommends giving the most important dashboard information the highest contrast and largest area, then stripping away non-essential information that distracts from interpretation: https://carbon-website-git-pictograms-docs.carbon-design-system.vercel.app/data-visualization/dashboards.
- Prefer “quiet hierarchy” over decorative cards: use section size, spacing, borders, muted labels, and one accent path for primary action; Carbon’s dashboard guidance emphasizes clear hierarchy and reduced complexity, and Atlassian’s grid guidance separates top-level layout from internal component spacing: https://carbon-website-git-pictograms-docs.carbon-design-system.vercel.app/data-visualization/dashboards, https://atlassian.design/foundations/grid.
- Treat color as comprehension support, not atmosphere: Atlassian says chart colors should use data-visualization tokens, a brand chart color is appropriate when only one data color is needed, and status/severity colors should be reserved for actual status/severity: https://atlassian.design/foundations/color/data-visualization-color.
- Keep data views consistent across the app: Carbon recommends consistent color assignments, consistent chart layout and spacing, and consistent legend positioning across dashboards: https://carbon-website-git-pictograms-docs.carbon-design-system.vercel.app/data-visualization/dashboards.
- Make the “Zoom-like” direction an influence, not a literal copy: use a calm blue/violet accent on neutral surfaces, but let Aplyx’s existing violet tokens remain the implementation source of truth for now; the local token file already defines neutral surfaces, violet accent, semantic colors, subtle borders, and a no-runtime-download default font stack: file:///Users/keshmuthu/aplyx/desktop/src/styles/tokens.css.
- Avoid rainbow dashboards and accent overuse: Atlassian recommends neutral chart color for less important information so highlighted data can stand out, and status colors should only represent status/severity: https://atlassian.design/foundations/color/data-visualization-color.

### Motion / Transition Recommendations
- Eliminate hard static page swaps by standardizing route transitions: MDN describes View Transitions as a way to reduce cognitive load, help users stay in context, and reduce perceived loading latency between states or views: https://developer.mozilla.org/en-US/docs/Web/API/View_Transition_API.
- Use View Transition API progressively where the desktop web runtime supports it; Chrome documents same-document transitions for single-page apps and says the API can transition between different views with CSS animation control: https://developer.chrome.com/docs/web-platform/view-transitions.
- Keep default route motion subtle: opacity plus 4–8px translate is appropriate for calm orientation because Apple flags scaling, spinning, and peripheral motion as common Reduced Motion triggers: https://developer.apple.com/help/app-store-connect/manage-app-accessibility/reduced-motion-evaluation-criteria/.
- Make motion reactive, direct, and context-appropriate: Microsoft’s Windows motion guidance says motion should provide feedback, reinforce spatial way-finding, and use consistent entry/dismiss behavior for surfaces that share the same entry point: https://learn.microsoft.com/en-us/windows/apps/design/signature-experiences/motion.
- Use short desktop timings: Windows standard animation durations include 83ms, 167ms, and 250ms, while Material says transitions must be fast enough not to cause waiting and that transitions beyond 400ms may feel too slow: https://learn.microsoft.com/en-us/windows/apps/design/motion/timing-and-easing, https://m1.material.io/motion/duration-easing.html.
- Reserve the slowest motion for full-page or large-surface changes: Material recommends longer durations only for large travel distances or dramatic surface-area changes, and shorter durations for minor changes: https://m1.material.io/motion/duration-easing.html.
- Keep accessibility nuanced: Apple says well-designed animations can convey information and should not be removed prematurely, but apps should support Reduced Motion for users affected by motion triggers: https://developer.apple.com/help/app-store-connect/manage-app-accessibility/reduced-motion-evaluation-criteria/.
- Implementation target: keep hover/focus/nav feedback near `--duration-quick`, route in/out near `--duration-normal`, and use `--duration-slow` only for large page or wizard transitions; the current app already has 120ms, 220ms, and 420ms tokens plus a Reduced Motion override to zero them out: file:///Users/keshmuthu/aplyx/desktop/src/styles/tokens.css.

### Future Font Directions
- Keep current fonts as the implementation default for now: Aplyx currently defaults to an offline system UI stack and offers a bundled Geist/Geist Mono preference without CDN fetches: file:///Users/keshmuthu/aplyx/desktop/src/styles/tokens.css, file:///Users/keshmuthu/aplyx/desktop/src/styles/base.css.
- Inter is the safest future dashboard candidate for dense product UI because its own documentation describes it as designed for detailed user interfaces, with tall x-height, text/display optical sizes, slashed zero, and tabular numbers: https://rsms.me/inter/.
- Geist is the strongest “premium developer-product” direction because Vercel describes it as designed for developers and designers with precision, clarity, functionality, and mono support; this aligns with Aplyx’s existing optional bundled Geist path: https://vercel.com/font, file:///Users/keshmuthu/aplyx/desktop/src/styles/tokens.css.
- IBM Plex is the strongest “institutional productivity” direction because IBM describes Plex as a broad corporate type family with Sans, Serif, Mono, and Condensed subfamilies designed for a global technology company: https://www.ibm.com/design/language/typography/typeface/.
- Atkinson Hyperlegible is the strongest accessibility-forward direction because Braille Institute says it uses distinctive letters and numbers to improve legibility and readability for low-vision readers: https://www.brailleinstitute.org/freefont/.
- For dashboards, require tabular numeric support or equivalent numeric alignment before adopting a future default; Inter explicitly documents tabular numbers, and dashboard tables/metrics depend on aligned figures for scanability: https://rsms.me/inter/.
- Avoid trendy display-first typefaces for the app body, narrow grotesks for dense tables, decorative serifs for metrics, and any font without strong numeral differentiation; Inter documents slashed zero and tabular numbers for disambiguation/alignment, and Atkinson Hyperlegible documents character differentiation as a core readability feature: https://rsms.me/inter/, https://www.brailleinstitute.org/freefont/.

## Conflicts
- View Transitions can reduce cognitive load during navigation, but motion can also trigger nausea, dizziness, headaches, or distraction for some users — https://developer.mozilla.org/en-US/docs/Web/API/View_Transition_API vs https://developer.apple.com/help/app-store-connect/manage-app-accessibility/reduced-motion-evaluation-criteria/. Resolution: use transitions for orientation, keep them opacity/translate based by default, avoid scale/spin/peripheral movement, and honor Reduced Motion.
- Windows standard animation durations are very short at 83ms, 167ms, and 250ms, while Material’s common transition guidance allows around 300ms and says large full-screen transitions may be longer but over 400ms may feel slow — https://learn.microsoft.com/en-us/windows/apps/design/motion/timing-and-easing vs https://m1.material.io/motion/duration-easing.html. Resolution: for a desktop productivity app, use 120–220ms for most interactions and reserve ~420ms only for rare large-surface transitions where continuity matters more than speed.
- Inter, Geist, IBM Plex, and Atkinson Hyperlegible each have a credible future case, but the app already has an offline system stack and bundled Geist preference — https://rsms.me/inter/, https://vercel.com/font, https://www.ibm.com/design/language/typography/typeface/, https://www.brailleinstitute.org/freefont/, file:///Users/keshmuthu/aplyx/desktop/src/styles/tokens.css. Resolution: do not change the default font in this plan; instead, treat future font work as a visual QA decision after layout and motion stabilize.

## Gaps
- Target runtime support for View Transition API inside the shipped desktop container is unverified; Chrome documents support in Chromium-based browsers, but the actual desktop runtime version must be tested in-app before depending on it: https://developer.chrome.com/docs/web-platform/view-transitions. To answer this, run a small desktop runtime probe for `document.startViewTransition` in development and release builds.
- No user-specific readability test has compared Aplyx’s current system stack, Geist, Inter, IBM Plex, and Atkinson Hyperlegible on the actual Jobs/Review/History/Resumes screens; each font source describes capabilities, but none validates Aplyx’s content density: https://rsms.me/inter/, https://vercel.com/font, https://www.ibm.com/design/language/typography/typeface/, https://www.brailleinstitute.org/freefont/. To answer this, create screenshot variants and review them on macOS and Windows at typical desktop sizes.
- No source here validates the exact emotional fit of “Zoom-like calm” for Aplyx’s job-application workflow; design-system sources support calm hierarchy, grid discipline, and restrained color, but not Aplyx-specific brand perception: https://atlassian.design/foundations/grid, https://carbon-website-git-pictograms-docs.carbon-design-system.vercel.app/data-visualization/dashboards. To answer this, collect qualitative feedback from the operator on 2–3 palette mockups.

## Recommendation
Build the desktop UI toward “quiet premium productivity”: preserve the existing shell/sidebar model, tighten screens into grid-aligned sections, reduce decorative accents, use neutral surfaces and borders for structure, reserve violet for primary action and selected state, and keep status colors semantic only. Standardize route changes around subtle fade/translate transitions immediately, then progressively add View Transition API/shared-element transitions for obvious continuity moments such as list row → detail, wizard step → review, and dashboard card expansion after runtime support is verified. Keep the current system stack as the default and Geist as the bundled preference; evaluate Inter as the likely future dense-dashboard default, Geist as the premium technical default, IBM Plex as the institutional/productivity variant, and Atkinson Hyperlegible as an accessibility option. This recommendation rests on the dashboard hierarchy findings, motion accessibility findings, and font capability findings above.

Weakest assumption: the app’s desktop runtime will support View Transition API reliably enough to make it the durable route-transition foundation; if that is false, the plan should use the existing CSS route-frame animation system as the baseline and add shared-element continuity manually only where it materially improves orientation.
